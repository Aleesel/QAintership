## Описание багов:


## Баг №1: Некорректное изображение в правом верхнем углу

*   **Описание:** В правом верхнем углу страницы отображается иконка-заглушка (изображение с листом бумаги), а не ожидаемая иконка или логотип.
*   **Приоритет:** Medium
*   **Обоснование:** Не ломает функционал, но портит интерфейс и доверие пользователя. Пользователь может подумать, что сайт сломан или это фишинговая страница.

---

## Баг №2: Несоответствие результатов поиска вводимому запросу и повторяющиеся фильтры данных по типу вакансии

*   **Описание:** В строке поиска введено слово "Аналитик", но в результатах поиска отображаются вакансии, не связанные с аналитикой (например, "Клиентский сервис", "Финансы, закупки и аудит"). Также в фильтрах активен фильтр "Data Science" и "Поддержка пользователей", который не соответствует поисковому запросу.
*   **Приоритет:** High
*   **Обоснование:** Основная функция поиска и фильтрации на сайте — находить релевантные вакансии. Если поиск выдает несоответствующие результаты, пользователь не сможет найти нужную информацию, что делает сайт бесполезным для его основной цели.

---

## Баг №3: Некорректное поведение фильтров

*   **Описание:** При активации фильтра "Руководящая роль" (который должен показывать только руководящие позиции) в результатах поиска все равно отображаются вакансии без указания "Руководитель" в названии (например, "Бизнес-аналитик бизнес-процессов поддержки").
*   **Приоритет:** High
*   **Обоснование:** Фильтр работает некорректно, что может ввести пользователя в заблуждение.

---

## Баг №4: Некорректное состояние выпадающего списка "Команда"

*   **Описание:** Выпадающий список "Команда" показывает "Вакансий пока нет", хотя в списке результатов есть вакансии, которые должны относиться к командам (например, "Руководитель Data Science в команду монетизации"). Это противоречие создает путаницу.
*   **Приоритет:** Medium
*   **Обоснование:** Эта ошибка вызывает путаницу у пользователя, так как он видит противоречивую информацию: с одной стороны, сообщение о том, что вакансий нет, с другой — реальные вакансии в списке.

---

## Баг №5: Некорректная работа фильтра по городу

*   **Описание:** В фильтре выбран город "Санкт-Петербург", но в результатах поиска отображаются вакансии с разными локациями ("Москва", "Самара"), что не соответствует выбранному фильтру.
*   **Приоритет:** Medium
*   **Обоснование:** Фильтр по локации работает некорректно, показывая не релевантные результаты. Это затрудняет поиск вакансий в конкретном городе.

## Баг №6: Некорректное отображение формата работы

*   **Описание:** В описании вакансий используются различные формулировки для указания формата работы: "можно удаленно", "удаленно", "офис и удаленно". Нужно привести форматы работы к единообразию.
*   **Приоритет:** Low
*   **Обоснование:** Ошибка снижает удобство использования сайта. Пользователю приходится внимательнее читать каждое описание, чтобы понять формат работы. Стандартизация формулировок улучшила бы UX.

# Задание 2 (вариант 1) - автоматизированное тестирование API

Проект содержит автоматизированные тесты для микросервиса объявлений Avito.

## Описание

Микросервис предоставляет REST API для работы с объявлениями:
- Создание объявлений
- Получение объявления по ID
- Получение всех объявлений продавца
- Получение статистики по объявлению

**Base URL**: `https://qa-internship.avito.com`


### 1. Запуск проекта

Для начала клонируйте проект себе в рабочую среду (я выполнял все в PyCharm)

Затем нужно установить модули для работы с тестами и http. Для этого в терминале папки проекта нужно запустить команду ниже:
```bash
pip install -r requirements.txt
```

### Запуск всех тестов

После установки модулей нужно запустить тестирование, так же из терминала выполнить команду ниже:
```bash
pytest
```


# Интерпретация результатов

Все тесты были успешно выполнены. Был обнаружен один баг с отрицательной ценой товара, о чем написано в [Баг-репорте](BUGS.md). 
Ниже распишу что делает каждый тест. 


## Раздел: GET /api/1/item/{id} — TestItem - получение одного товара по UUID товара

### TC-1 — Получение существующего товара

*   **Описание:** Запрос заранее созданного товара по валидному UUID.
*   **Что проверяет:**
    *   Корректное получение объекта
    *   Валидность схемы ответа
    *   Что товар в ответе совпадает с запрошенным из эталонных данных в папке test_data
*   **Ожидаемый результат:** `200 OK`, корректный JSON, схема валидна.

### TC-2 — Получение несуществующего товара

*   **Описание:** Запрос товара по UUID, который отсутствует в базе.
*   **Что проверяет:** Корректную обработку отсутствующего товара
*   **Ожидаемый результат:** `404 Not Found`.

### TC-3 — Передача невалидного ID товара

*   **Описание:** Запрос товара с ID неверного типа (число вместо UUID).
*   **Что проверяет:** Валидацию параметра `id`
*   **Ожидаемый результат:** `400 Bad Request`.

## Раздел: GET /api/1/{sellerId}/item — TestItems - получение товаров по sellerId

### TC-4 — Получение товаров по валидному sellerId

*   **Описание:** Запрос списка товаров существующего продавца.
*   **Что проверяет:**
    *   Возврат всех корректных товаров для `sellerId`
    *   Валидность структуры ответа
*   **Ожидаемый результат:** `200 OK`, список товаров продавца.

### TC-5 — Получение товаров по несуществующему sellerId

*   **Описание:** Запрос товаров по `sellerId`, которого нет в системе.
*   **Что проверяет:** Обработку отсутствующих данных
*   **Ожидаемый результат:** `200 OK`, пустой список.

### TC-6 — Передача невалидного sellerId

*   **Описание:** Передача `sellerId` неправильного формата (string вместо числа).
*   **Что проверяет:** Корректность серверной валидации
*   **Ожидаемый результат:** `400 Bad Request`.

## Раздел: POST /api/1/item (+ DELETE /api/2/item/{id}) — TestPost - проверка создания товара 

### TC-10 — Создание товара с пустым телом

*   **Описание:** POST запрос с `{}` в теле.
*   **Что проверяет:**
    *   Валидацию обязательных полей
    *   Схему ответа сервера
*   **Ожидаемый результат:** `400 Bad Request`.

### TC-11 — Создание товара с валидным телом + удаление

*   **Описание:** Создание товара со всеми полями и последующий DELETE.

### TC-12 — Создание товара с невалидным JSON

*   **Ожидаемый результат:**
    *   POST → `200 OK` (фактическое поведение сервиса)

*   **Описание:** POST с value `price`, выходящим за границы `int`.
*   **Что проверяет:** Серверную числовую валидацию
*   **Ожидаемый результат:** `400 Bad Request`.
